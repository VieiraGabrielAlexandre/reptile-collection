## Official Database Strategy

Use:

* PostgreSQL;
* versioned migrations;
* explicit constraints;
* explicit indexes;
* parameterized SQL;
* `sqlc`;
* transactions when required;
* integration tests against PostgreSQL.

Do not use an ORM as the default persistence strategy.

Do not rely on application validation alone when the database can enforce a critical invariant safely.

Do not use LocalStack to emulate PostgreSQL.

---

## Modeling Principles

The schema must prioritize:

* data integrity;
* explicit meaning;
* predictable queries;
* evolvability;
* appropriate normalization;
* useful constraints;
* clear ownership;
* safe migrations;
* observable performance.

Avoid both extremes:

* one oversized table containing every concept;
* excessive fragmentation into dozens of tiny tables without operational value.

Model according to actual domain behavior.

---

## Schema Ownership

Each module owns its primary tables.

Examples:

```text
taxonomy -> taxa, taxonomy_relationships
species -> species, species_common_names, species_taxa
articles -> articles, article_revisions, article_species
media -> media_assets
users -> users, roles, user_roles
gamification -> activity_events
```

Shared foreign keys do not remove module ownership.

Do not create generic tables such as:

```text
entities
metadata
objects
records
items
```

unless the domain genuinely requires them.

---

## Naming Conventions

Use lowercase snake_case.

Examples:

```text
species
scientific_name
published_at
article_revisions
media_assets
```

Use plural table names.

Use explicit foreign key names.

Examples:

```text
author_id
species_id
article_id
primary_image_id
```

Avoid vague names:

```text
type
data
info
value
status_id
ref
```

unless the surrounding context makes the meaning unambiguous.

---

## Primary Keys

Use a consistent ID strategy.

Preferred default:

* UUID generated by the application or database;
* stored as PostgreSQL `uuid`.

Example:

```sql
id uuid PRIMARY KEY
```

If the database generates IDs:

```sql
id uuid PRIMARY KEY DEFAULT gen_random_uuid()
```

Ensure the required extension is created intentionally.

Do not mix integer IDs and UUIDs arbitrarily across core domain tables.

Do not expose sequential IDs publicly without a documented reason.

---

## Timestamps

Core mutable entities should generally include:

```sql
created_at timestamptz NOT NULL
updated_at timestamptz NOT NULL
```

Published resources may also include:

```sql
published_at timestamptz
scheduled_at timestamptz
archived_at timestamptz
```

Use `timestamptz`.

Store timestamps in UTC semantics.

Do not use `timestamp without time zone` for application events without an explicit reason.

Do not add every possible lifecycle timestamp preemptively.

---

## Defaults

Use defaults only when they represent safe and universal behavior.

Appropriate examples:

```sql
created_at timestamptz NOT NULL DEFAULT now()
updated_at timestamptz NOT NULL DEFAULT now()
```

Potentially risky examples:

```sql
status text NOT NULL DEFAULT 'published'
is_admin boolean NOT NULL DEFAULT true
```

Do not use defaults that hide missing application decisions.

---

## Nullability

Use `NOT NULL` when absence is not a valid state.

Allow `NULL` when:

* the value is truly optional;
* the lifecycle permits absence;
* the concept is unknown;
* publication has not occurred;
* no relationship exists.

Do not use empty strings as substitutes for missing values.

Do not mark every column nullable merely to simplify inserts.

---

## Enumerations

Prefer one of these strategies:

### Check constraint

Useful for stable, small sets:

```sql
status text NOT NULL
  CHECK (status IN ('draft', 'in_review', 'scheduled', 'published', 'archived'))
```

### Lookup table

Useful when values have:

* metadata;
* lifecycle;
* localization;
* administrative management.

### PostgreSQL enum

Use cautiously.

PostgreSQL enums are acceptable for highly stable values, but schema evolution can be less flexible.

Default preference: text plus check constraint for editorial workflow states.

Do not use magic strings without database validation for critical statuses.

---

## Domain Constraints

Use constraints for invariants that must remain true regardless of application path.

Examples:

```sql
CHECK (minimum_value IS NULL OR maximum_value IS NULL OR minimum_value <= maximum_value)
```

```sql
CHECK (
  status <> 'scheduled'
  OR scheduled_at IS NOT NULL
)
```

```sql
UNIQUE (slug)
```

Use database constraints for:

* uniqueness;
* required relationships;
* valid ranges;
* allowed status values;
* non-self references where practical;
* referential integrity.

Do not place highly contextual business workflows entirely in database triggers.

---

## Unique Constraints

Use unique constraints intentionally.

Examples:

```sql
UNIQUE (slug)
UNIQUE (scientific_name)
UNIQUE (species_id, normalized_name, language, region)
UNIQUE (user_id, role_id)
```

Before creating uniqueness, determine whether scope matters.

Examples:

* article slugs may be unique globally;
* common names may only be unique per species, locale, and region;
* taxonomic names may require synonym support.

Do not add global uniqueness based on assumptions.

---

## Foreign Keys

Use foreign keys to preserve referential integrity.

Example:

```sql
author_id uuid NOT NULL REFERENCES users(id)
```

Choose deletion behavior deliberately.

Possible behaviors:

```sql
ON DELETE RESTRICT
ON DELETE CASCADE
ON DELETE SET NULL
```

Guidance:

* use `RESTRICT` when deletion would invalidate important history;
* use `CASCADE` for dependent join rows;
* use `SET NULL` when preserving the child is valid without the parent.

Do not apply `CASCADE` automatically.

Do not create orphanable relationships without a reason.

---

## Deletion Strategy

Do not add soft delete to every table.

Use hard delete when:

* the record is disposable;
* no audit or recovery requirement exists;
* referential integrity is handled.

Use archival or status transitions when:

* the resource is editorial;
* URLs must remain stable;
* history matters;
* publication lifecycle exists.

Potential examples:

* articles use `archived`;
* species may use editorial status;
* temporary tokens may be hard deleted;
* join rows may cascade.

Do not mix `deleted_at` and editorial status without a clear lifecycle model.

---

## Taxonomy Modeling

Scientific taxonomy must remain distinct from editorial grouping.

A flexible initial taxonomy model may use:

```sql
CREATE TABLE taxa (
  id uuid PRIMARY KEY,
  rank text NOT NULL,
  scientific_name text NOT NULL,
  parent_id uuid REFERENCES taxa(id),
  status text NOT NULL,
  accepted_taxon_id uuid REFERENCES taxa(id),
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);
```

Potential constraints:

```sql
CHECK (id <> parent_id)
CHECK (id <> accepted_taxon_id)
```

Do not implement a generalized biological ontology unless required.

For an initial phase, a simpler schema may be preferable.

The chosen model must support actual project requirements, not theoretical completeness.

---

## Species Modeling

A minimal species table may contain:

```sql
CREATE TABLE species (
  id uuid PRIMARY KEY,
  slug text NOT NULL UNIQUE,
  scientific_name text NOT NULL,
  preferred_common_name text NOT NULL,
  summary text,
  editorial_group text,
  editorial_status text NOT NULL,
  author_id uuid,
  primary_image_id uuid,
  published_at timestamptz,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);
```

This is directional, not mandatory.

Evaluate whether scientific name uniqueness must account for:

* accepted names;
* synonyms;
* subspecies;
* disputed taxonomy.

Do not place all habitat, behavior, diet, conservation, and references in the same initial table without evaluating their cardinality.

---

## Common Names

Common names should use a related table when locale and regional metadata matter.

Example:

```sql
CREATE TABLE species_common_names (
  id uuid PRIMARY KEY,
  species_id uuid NOT NULL REFERENCES species(id) ON DELETE CASCADE,
  name text NOT NULL,
  normalized_name text NOT NULL,
  language_code text NOT NULL,
  region_code text,
  is_preferred boolean NOT NULL DEFAULT false,
  source_reference_id uuid,
  created_at timestamptz NOT NULL DEFAULT now()
);
```

Potential uniqueness:

```sql
UNIQUE (
  species_id,
  normalized_name,
  language_code,
  COALESCE(region_code, '')
)
```

PostgreSQL constraints involving expressions may require a unique index instead.

Do not store common names as comma-separated values.

## Articles

A minimal articles table may include:

```sql
CREATE TABLE articles (
  id uuid PRIMARY KEY,
  slug text NOT NULL UNIQUE,
  title text NOT NULL,
  subtitle text,
  summary text,
  content jsonb NOT NULL,
  status text NOT NULL,
  author_id uuid NOT NULL REFERENCES users(id),
  cover_image_id uuid REFERENCES media_assets(id),
  estimated_reading_minutes integer,
  scheduled_at timestamptz,
  published_at timestamptz,
  archived_at timestamptz,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);
```

Use JSONB for structured article blocks only if the structure is validated in the application and supported by tests.

Do not use JSONB to avoid modeling core searchable metadata.

---

## JSONB Guidance

Use JSONB when:

* content is structured but evolves;
* the value is loaded as one unit;
* schema flexibility is beneficial;
* relational joins would add little value;
* application validation is explicit.

Appropriate example:

* structured article blocks.

Less appropriate examples:

* author IDs;
* publication status;
* taxonomy;
* frequently filtered conservation status;
* tags requiring joins;
* permissions.

Every JSONB document should have:

* explicit application schema;
* versioning strategy when necessary;
* validation tests;
* safe parsing;
* migration strategy for incompatible shape changes.

Do not accept arbitrary JSON objects.

## References

References may be modeled independently.

Example:

```sql
CREATE TABLE references (
  id uuid PRIMARY KEY,
  title text NOT NULL,
  authors text,
  publication text,
  publication_year integer,
  publisher text,
  doi text,
  url text,
  accessed_at date,
  reference_type text NOT NULL,
  language_code text,
  notes text,
  created_at timestamptz NOT NULL DEFAULT now()
);
```

Relationships may include:

```text
article_references
species_references
conservation_assessment_references
```

Do not duplicate the same reference row for every resource when reuse is useful.

Do not over-engineer claim-level citations in the initial phase.

## Conservation Assessments

Conservation data should support source and date.

Example:

```sql
CREATE TABLE conservation_assessments (
  id uuid PRIMARY KEY,
  species_id uuid NOT NULL REFERENCES species(id) ON DELETE CASCADE,
  authority text NOT NULL,
  category text NOT NULL,
  scope text,
  assessment_date date,
  publication_year integer,
  population_trend text,
  source_url text,
  notes text,
  created_at timestamptz NOT NULL DEFAULT now()
);
```

Potential uniqueness must be considered carefully.

An authority may publish multiple assessments over time.

Do not overwrite historical assessments without a product decision.

---

## Join Tables

Use explicit join tables for many-to-many relationships.

Examples:

```text
article_species
article_tags
species_tags
user_roles
species_habitats
species_ecological_roles
```

A join table should usually contain:

* both foreign keys;
* a composite primary key or unique constraint;
* metadata only when the relationship has meaning.

Example:

```sql
CREATE TABLE article_species (
  article_id uuid NOT NULL REFERENCES articles(id) ON DELETE CASCADE,
  species_id uuid NOT NULL REFERENCES species(id) ON DELETE CASCADE,
  relationship_type text,
  created_at timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (article_id, species_id)
);
```

Do not create an independent surrogate ID for every pure join table without a reason.

---

## Indexing

Create indexes based on actual query patterns.

Common index candidates:

* slugs;
* foreign keys;
* status plus publication date;
* normalized search fields;
* join-table reverse lookups;
* full-text search vectors;
* scheduled publication dates;
* event idempotency keys.

Example:

```sql
CREATE INDEX idx_articles_published
  ON articles (published_at DESC)
  WHERE status = 'published';
```

Example:

```sql
CREATE INDEX idx_species_editorial_group
  ON species (editorial_group)
  WHERE editorial_status = 'published';
```

Do not index every column.

Every index has write and storage cost.

Document non-obvious indexes.

---

## Foreign Key Indexes

PostgreSQL does not automatically create indexes for referencing foreign keys.

Evaluate indexes for columns such as:

```text
author_id
species_id
article_id
role_id
primary_image_id
```

Create them when query or deletion behavior benefits.

Do not add duplicate indexes already covered by composite indexes.

---

## Partial Indexes

Use partial indexes when queries consistently target a subset.

Examples:

```sql
CREATE INDEX idx_articles_scheduled
  ON articles (scheduled_at)
  WHERE status = 'scheduled';
```

```sql
CREATE INDEX idx_species_public_slug
  ON species (slug)
  WHERE editorial_status = 'published';
```

Confirm whether uniqueness must apply globally or only to a state subset.

Avoid partial indexes when the query planner cannot use them reliably due to query shape.

---

## Full-Text Search

The initial search strategy may use PostgreSQL Full Text Search.

Possible approach:

* generated or maintained `tsvector`;
* language configuration;
* weighted fields;
* GIN indexes.

Example concept:

```sql
search_vector tsvector GENERATED ALWAYS AS (
  setweight(to_tsvector('simple', coalesce(preferred_common_name, '')), 'A') ||
  setweight(to_tsvector('simple', coalesce(scientific_name, '')), 'A') ||
  setweight(to_tsvector('simple', coalesce(summary, '')), 'B')
) STORED
```

Then:

```sql
CREATE INDEX idx_species_search_vector
  ON species USING gin(search_vector);
```

Language choice must reflect content strategy.

Do not blindly use `english` if content may be Portuguese or multilingual.

A future localized strategy may require separate vectors or language-aware queries.

---

## Query Design

Queries must be:

* explicit;
* parameterized;
* bounded;
* deterministic;
* readable;
* compatible with sqlc;
* aligned with indexes.

Prefer named SQL files grouped by module.

Example:

```sql
-- name: FindSpeciesBySlug :one
SELECT
  id,
  slug,
  scientific_name,
  preferred_common_name,
  summary,
  editorial_group,
  published_at
FROM species
WHERE slug = $1
  AND editorial_status = 'published';
```

Do not use `SELECT *`.

Explicit columns protect contracts and mappings.

---

## Pagination

Queries must enforce bounded pagination.

Example:

```sql
-- name: ListPublishedSpecies :many
SELECT
  id,
  slug,
  scientific_name,
  preferred_common_name,
  summary
FROM species
WHERE editorial_status = 'published'
ORDER BY preferred_common_name ASC, id ASC
LIMIT $1
OFFSET $2;
```

Use deterministic ordering.

For large tables or unstable datasets, cursor pagination may be preferable.

Do not allow unbounded list queries.

---

## Dynamic Filters

Do not build SQL by concatenating user input.

For a small set of optional filters, use explicit query variants or safe conditional clauses.

Example:

```sql
WHERE
  editorial_status = 'published'
  AND ($1::text IS NULL OR editorial_group = $1)
```

Evaluate planner impact.

For complex filters, controlled query builders may be considered in infrastructure, but all identifiers and operators must be whitelisted.

Do not pass arbitrary sort columns directly from the API.

---

## sqlc

Use sqlc to generate typed query code.

The configuration should define:

* PostgreSQL engine;
* schema paths;
* query paths;
* Go package output;
* database package;
* type overrides when needed.

Example:

```yaml
version: "2"
sql:
  - engine: "postgresql"
    schema: "migrations"
    queries: "queries"
    gen:
      go:
        package: "db"
        out: "internal/platform/database/generated"
        sql_package: "pgx/v5"
```

Exact configuration may differ.

Do not manually edit sqlc-generated files.

Run generation after relevant SQL changes.

---

## Type Overrides

Use sqlc overrides when domain compatibility benefits.

Possible examples:

* UUID to `github.com/google/uuid.UUID`;
* `timestamptz` to `time.Time`;
* nullable types to pgx types or pointers.

Do not create complicated custom overrides without clear value.

Generated database types are infrastructure types, not domain entities.

## Migrations

Every schema change requires a versioned migration.

Migration naming should be consistent.

Example:

```text
000001_create_users.up.sql
000001_create_users.down.sql
000002_create_species.up.sql
000002_create_species.down.sql
```

Or timestamp-based names if the project adopts that strategy.

Do not mix migration naming conventions.

Each migration should be focused and understandable.

---

## Migration Rules

Migrations must:

* be versioned;
* be deterministic;
* avoid environment-specific assumptions;
* preserve data when required;
* include rollback when safe;
* use explicit transactions when supported;
* be tested from a clean database;
* be tested against an existing schema when compatibility matters.

Do not edit a migration already applied in a shared environment.

Create a new corrective migration.

---

## Up and Down Migrations

Down migrations should exist when rollback is safe and meaningful.

Do not create destructive down migrations that imply safe rollback when data cannot be restored.

For irreversible changes, document the limitation explicitly.

Example:

```sql
-- Irreversible data migration.
-- Down migration intentionally fails.
```

The exact project migration tool may have a preferred convention.

Do not pretend data-destructive operations are reversible.

---

## Safe Migration Strategy

For production-oriented changes, prefer expand-and-contract.

Example:

1. add a nullable column;
2. deploy code that writes both old and new formats;
3. backfill data;
4. validate;
5. enforce `NOT NULL`;
6. remove old column in a later release.

Do not perform risky schema rewrites in one step when they could lock large tables or break running code.

The local-only phase may permit simpler migrations, but design should remain evolvable.

---

## Data Backfills

Backfills must be:

* explicit;
* idempotent when possible;
* observable;
* bounded;
* safe for partial failure.

Do not hide large backfills inside an application startup.

For small seed-like corrections, a migration may be acceptable.

For large future datasets, use dedicated commands or jobs.

---

## Seeds

Seed data exists for:

* local development;
* demonstrations;
* integration tests;
* reproducible environments.

Seed data must be:

* deterministic;
* safe to rerun;
* clearly non-production;
* representative;
* free from copyrighted or unlicensed media;
* free from sensitive data.

Include a small number of realistic reptile records.

Do not invent scientific facts casually.

Use clearly documented sample data or references.

---

## Seed IDs

Use fixed IDs for deterministic seed relationships when helpful.

Example:

```text
00000000-0000-0000-0000-000000000101
```

Do not rely on insertion order to recover generated IDs.

Ensure local-only identifiers cannot be confused with production records when that matters.

---

## Transactions

Use transactions for operations requiring atomicity.

Examples:

* article plus initial revision;
* species plus common names;
* role assignment plus audit event;
* publication plus status history;
* media metadata plus ownership relationship.

Do not use a transaction for unrelated reads.

Do not keep a transaction open during external service calls.

Do not nest transactions unless the chosen library and strategy explicitly support it.

---

## Isolation and Locking

Use PostgreSQL defaults unless a use case requires stronger guarantees.

Evaluate concurrency for:

* slug uniqueness;
* scheduled publication;
* revision numbering;
* idempotent activity events;
* role assignment;
* duplicate imports.

Prefer unique constraints and atomic SQL over application-only checks.

Example:

Do not rely only on:

```text
SELECT whether slug exists
then INSERT
```

The database must still enforce uniqueness.

---

## Optimistic Concurrency

For editing workflows, optimistic concurrency may use:

* `updated_at`;
* integer version column;
* revision ID;
* ETag-compatible version.

Example:

```sql
UPDATE articles
SET
  title = $2,
  updated_at = now(),
  version = version + 1
WHERE id = $1
  AND version = $3;
```

Check affected row count.

Do not implement optimistic locking before concurrent editing becomes a requirement.

Article revisions may already provide a more appropriate mechanism.

---

## Article Revisions

A revision table may include:

```sql
CREATE TABLE article_revisions (
  id uuid PRIMARY KEY,
  article_id uuid NOT NULL REFERENCES articles(id) ON DELETE CASCADE,
  revision_number integer NOT NULL,
  title text NOT NULL,
  subtitle text,
  summary text,
  content jsonb NOT NULL,
  created_by uuid NOT NULL REFERENCES users(id),
  created_at timestamptz NOT NULL DEFAULT now(),
  UNIQUE (article_id, revision_number)
);
```

Revision numbering must be safe under concurrency.

Possible strategies:

* transaction with row lock;
* database sequence per article alternative;
* generated revision UUID without sequential display requirement.

Do not create revisions unless Phase 3 requires them.

---

## Idempotency

Database constraints may support idempotency.

Example:

```sql
idempotency_key text NOT NULL UNIQUE
```

Useful for:

* activity events;
* imported records;
* queue processing;
* callback handling.

Idempotency keys should have clear scope.

Example:

```sql
UNIQUE (user_id, event_type, idempotency_key)
```

Do not add idempotency columns to every table.

---

## Audit Data

Audit requirements may include:

* who created;
* who updated;
* who published;
* when;
* previous revision;
* reason.

Use explicit audit tables when history matters.

Do not overload application logs as the only audit record.

Do not store full sensitive request payloads in audit tables.

---

## Repository Mapping

Repository implementations should map between generated SQL rows and domain types.

Do not expose generated sqlc models directly to domain or transport layers unless deliberately accepted.

Mapping must handle:

* nullable fields;
* UUIDs;
* timestamps;
* enums;
* JSONB;
* missing rows;
* database errors.

Example error translation:

```go
if errors.Is(err, pgx.ErrNoRows) {
    return nil, species.ErrNotFound
}
```

Unique violations should map to domain or application conflict errors.

Do not leak raw PostgreSQL errors to clients.

---

## PostgreSQL Error Mapping

Common SQLSTATE values may be used carefully.

Examples:

```text
23505 unique_violation
23503 foreign_key_violation
23514 check_violation
40001 serialization_failure
40P01 deadlock_detected
```

Map only known cases.

Wrap unknown errors with context and treat them as internal.

Do not compare database error messages as strings.

---

## Query Timeouts

Database operations must respect context deadlines.

Configure pool and query behavior intentionally.

Potential settings:

* maximum connections;
* minimum connections;
* maximum connection lifetime;
* idle timeout;
* health check period.

Do not use overly large connection pools by default.

Connection limits must reflect local and future RDS capacity.

---

## Readiness

Database readiness should verify connectivity with a short timeout.

Use a lightweight query such as:

```sql
SELECT 1;
```

Do not run migrations or expensive checks in readiness endpoints.

Do not expose database connection details.

---

## Performance Review

Before optimizing, inspect:

* query plans;
* row estimates;
* sequential scans;
* missing indexes;
* join cardinality;
* sorting;
* pagination;
* repeated queries;
* N+1 patterns.

Use:

```sql
EXPLAIN
EXPLAIN ANALYZE
```

Use `EXPLAIN ANALYZE` only in safe environments because it executes the query.

Do not add indexes solely because a column appears in a filter.

Validate the actual query plan.

---

## N+1 Prevention

Avoid loading related data one row at a time.

Possible strategies:

* joins;
* batched `WHERE id = ANY($1)`;
* aggregate queries;
* dedicated detail queries;
* application-level batching.

Do not create one giant join that duplicates excessive data without evaluating payload and mapping complexity.

Choose query shapes per use case.

---

## Counts

Pagination counts may be expensive.

For initial datasets, exact counts are acceptable.

For large future datasets, consider:

* separate count queries;
* optional totals;
* cached estimates;
* cursor pagination without total count.

Do not prematurely complicate counts.

---

## Data Import

Future imports must validate:

* taxonomy;
* scientific names;
* duplicates;
* references;
* units;
* locale;
* source attribution;
* row-level errors.

Imports should support dry-run and clear error reporting when introduced.

Do not directly insert untrusted CSV data without validation.

Do not implement import pipelines before the relevant phase or requirement.

---

## Security

Database changes must evaluate:

* SQL injection;
* least privilege;
* sensitive columns;
* accidental public exposure;
* tenant or object-level authorization implications;
* data retention;
* backup requirements;
* destructive operations;
* auditability.

Use parameterized queries.

Do not store plaintext passwords.

Do not store raw access tokens unless explicitly required and protected.

Do not grant the application superuser privileges.

## Database Roles

Future deployment should separate responsibilities where practical.

Possible roles:

* migration role;
* application runtime role;
* read-only operational role.

Local development may use a simpler role model.

Do not require production-level role complexity in Phase 0 unless it blocks future safety.

Document privilege assumptions.

---

## Sensitive Data

The platform should minimize personal data.

Potential user data:

* external identity ID;
* email;
* display name;
* roles;
* account status;
* activity history.

Do not store identity-provider passwords.

Do not store unnecessary profile data.

Evaluate retention and deletion requirements before expanding personal data.

## Testing Strategy

Database behavior requires integration tests against PostgreSQL.

Test:

* migrations from empty database;
* migration order;
* constraints;
* uniqueness;
* foreign keys;
* check constraints;
* queries;
* pagination;
* filtering;
* repository mapping;
* transactions;
* conflict handling;
* rollback behavior when relevant.

Do not mock SQL for repository behavior when a real PostgreSQL test is practical.

---

## Test Database

Use an isolated PostgreSQL instance.

Possible strategies:

* Testcontainers;
* dedicated Docker Compose test service;
* temporary database per test suite.

Tests must not depend on the developer's manual database state.

Tests must clean up or use isolated transactions/databases.

Do not point tests at production-like shared credentials.

---

## Migration Tests

At minimum, validate:

```text
empty database -> latest schema
```

When compatibility matters, also validate:

```text
previous schema -> latest schema
```

Verify important constraints with failing inserts.

Do not assume a migration is valid because the SQL parses.

---

## Seed Tests

Verify that seed execution:

* succeeds;
* can be rerun if designed as idempotent;
* creates expected core records;
* preserves foreign keys;
* does not create duplicates.

Do not require seed data for unit tests.

---

## sqlc Validation

After SQL changes:

```bash
sqlc generate
```

Then run:

```bash
go test ./...
go build ./...
```

Generated code must compile.

Do not commit SQL that causes stale generated code unless repository policy explicitly generates only in CI.

## Migration Commands

Use the migration tool chosen by the repository.

Expected Makefile commands may include:

```bash
make migrate
make migrate-down
make migrate-create name=create_species
make migration-status
```

Do not require developers to remember long raw commands.

Document required environment variables.

---

## Expected Validation Commands

From the repository root or backend directory, as applicable:

```bash
docker compose config
make migrate
make seed
make test
```

Database-specific checks:

```bash
sqlc generate
go test ./...
```

Migration reset in local or test environment:

```bash
make reset
make migrate
make seed
```

Never run reset commands against a real or shared environment without explicit authorization.

Only report commands actually executed.

## Implementation Workflow

When using this skill:

1. identify the owning module;
2. inspect the current schema;
3. identify domain invariants;
4. define cardinality;
5. define nullability;
6. define constraints;
7. define indexes based on queries;
8. create the migration;
9. update SQL queries;
10. regenerate sqlc;
11. update repositories;
12. update seeds;
13. add integration tests;
14. run migration and query validation;
15. update documentation.

Implement the smallest schema change that supports the current use case.

## Definition of Done

A database task is complete only when:

* the schema represents the required domain behavior;
* migration files exist;
* migrations apply successfully;
* rollback is tested when safe and supported;
* constraints protect critical invariants;
* indexes support known queries;
* SQL is parameterized;
* sqlc generation succeeds;
* generated code compiles;
* repository mappings are updated;
* seeds are updated when necessary;
* integration tests pass;
* documentation is updated;
* no destructive operation is hidden;
* no validation result is falsely claimed.

---

## Prohibited Practices

Do not:

* use `SELECT *`;
* concatenate untrusted SQL;
* expose raw database errors;
* edit applied migrations;
* rely only on application validation for critical integrity;
* use JSONB to avoid modeling everything;
* add indexes to every column;
* add soft delete to every table;
* use cascade deletion without analysis;
* store multiple values as comma-separated text;
* use empty strings as null replacements;
* store plaintext passwords;
* give the runtime application superuser access;
* create unbounded list queries;
* build arbitrary sorting from client input;
* run destructive resets against shared environments;
* manually edit generated sqlc code;
* create tables for future phases without a current requirement;
* declare completion without applying migrations and running tests.
